# /id8 Workflow

Invocation: `/id8`

Follow the sequence exactly. Never combine multiple steps into one response.

## Strict Step Sequence
1. Generate a PRD from the initial prompt. Ask the user for a project title, or suggest one if not provided.
   Pause after output. Ask for confirmation before continuing. Save PRD to `prd.md`.
2. Call Context7 and produce a technical plan compatible with Vercel. Avoid using services that require outside API keys, use keyless APIs if needed.
   Include version-sensitive implementation rules in the plan when relevant (for example Next.js 16 `proxy.ts`, Tailwind v4 font loading rules, and TypeScript transition-safe patterns).
   If the project requires a backend or database, include Supabase in the technical plan and specify the required tables, auth, RLS policies, and edge functions.
   Pause after output. Ask for confirmation before continuing. Save technical plan to `tech_plan.md`.
3. Ask the user if they want the design to be generated by AI or not.
   If yes, ask what device type to design for and which model to use (`Gemini 3 flash` or `Gemini 3 pro`), then use Stitch MCP with strict parameter contracts:
   `create_project` accepts `title` only.
   `generate_screen_from_text` must use `projectId` as the bare numeric ID (no `projects/` prefix). `deviceType` belongs here, not in `create_project`.
   `list_screens` must use bare `projectId` (not `parent`), only run at the end of the design generation process and at least 3 minutes after screen generation.
   `get_screen` may use full resource names like `projects/{project}/screens/{screen}`.
   Generation takes up to 4 minutes, if a generation call returns no output, treat it as "possibly pending" and continue.
   Pause after user feedback loop finishes.
   Provide the Stitch project link using the format `https://stitch.withgoogle.com/projects/projectID` (replace `projectID` with the actual project ID).
   Ask the user to export the screens they like to MCP and paste the exported code in the chat.
   Use the user-pasted exported screen code as the source of truth for implementation in following steps. Save exported screen data to `screens` folder.
4. The installer has already scaffolded `id8-src/` (Next.js app with recommended defaults) in the project directory.
   If the tech plan includes Supabase, initialize the Supabase project:
   Run `supabase init` in the project directory (if not already initialized).
   Run `supabase link --project-ref <ref>` to link to an existing Supabase project, or ask the user to create one via `supabase projects create <name>`.
   Apply any required migrations with `supabase db push`.
   Continue
5. Implement frontend code in the `id8-src/` folder using approved screens, the PRD, the tech plan, and Context7 references.
   if generated, use screen data downloaded in the `screens` folder as context.
   Ensure the frontend matches the approved design and fully implements PRD features. If any features in the PRD or Tech Plan are not in the designed screens, follow the design language to implement them.
   If using Supabase, configure the Supabase client in the frontend and implement backend interactions (auth, database queries, edge functions) as specified in the tech plan.
   Do not pause for confirmation during implementation unless user review is needed.
   Continue
6. Run the frontend locally from `id8-src/` and fix issues until clean.
   Run both development and production-grade checks (build/typecheck) to catch deploy-only failures.
   Apply framework guardrails during fixes:
   For Next.js 16+, use `proxy.ts` (not `middleware.ts`) and export `proxy`.
   For React `startTransition`, ensure callback returns `void` (wrap async actions as `async () => { await ... }`).
   For Tailwind v4, load external fonts via `<link>` in `layout.tsx` head, not CSS `@import url(...)`.
   If using Material Symbols, ensure the `<link>` specifies all variable axes (opsz,wght,FILL,GRAD) and the CSS class contains all required font rendering properties.
   Run until no errors, then continue.
7. Create a `.gitignore` file to exclude unnecessary files.
   Run a security check for secrets/keys before any push; if secrets are detected, stop and notify the user.
   Check if any nested `.git` directories exist (e.g. `id8-src/.git`) and remove them to prevent them from being treated as git submodules.
   Check if the current directory is already in a Git repository. If not, initialize one with `git init`.
   Use the GitHub CLI to create the repository: `gh repo create <name> --public --source=. --remote=origin`.
   Push code with local git commands: `git add .`, `git commit -m "Initial commit"`, `git push -u origin main`.
   Mandatory confirmation required immediately before any repo creation/push action.
8. Deploy frontend to Vercel using the Vercel CLI. Make Vercel site name match project title (must be lowercase).
   Link the project: `npx vercel link --yes --project <project-name>` in the `id8-src/` directory. If it links to the wrong project, delete `.vercel/` and re-link.
   Connect to GitHub for automatic deploys: `npx vercel git connect`.
   Set environment variables if needed: `npx vercel env add <name> <environment>`.
   If using Supabase, ensure `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` are set as Vercel environment variables.
   Deploy to production: `npx vercel deploy --prod --yes`.
   Keep secrets out of logs/outputs.
   Mandatory confirmation required immediately before deployment.
9. Perform post-deploy verification and publish a completion report with links, status, and outstanding actions.
   Verify core user flows plus production-only checks (auth/session behavior, icon/font rendering, and env wiring).
   Pause after final report for explicit closeout confirmation.

## Dry-Run Mode
When `--dry-run` is present:

1. Validate Context7 connectivity with a read-only docs lookup.
2. Validate Stitch connectivity with a read-only project/list call.
3. Verify GitHub CLI is authenticated: `gh auth status`.
4. Verify Vercel CLI is authenticated: `npx vercel whoami`.
5. Verify Supabase CLI is authenticated: `supabase projects list` (if Supabase is in the tech plan).
6. Stop after reporting connectivity status. Do not create repos, deploy, or provision resources.

## Confirmation Gates
Required confirmation prompts:

1. Before Step 7, ask for explicit approval to create/push repositories.
2. Before Step 8, ask for explicit approval to deploy frontend to Vercel.
3. If approval is denied, stop and report what was skipped.

## Execution Rules

1. Do not hallucinate tool output.
2. Do not run Steps 7 or 8 without explicit approval.
3. If a required MCP server is not authenticated, stop and report the required login/action.
4. If `--dry-run` is set, perform connectivity checks only.
5. Before each MCP call, validate parameter names and types against the current tool schema.
